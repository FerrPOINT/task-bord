# Модели и сервисы

Архитектура веб-приложения в общем случае делится на две части: модели и сервисы.

Сервисы обрабатывают все "сырые" запросы, модели содержат данные и методы для работы с ними.

Сервис обычно принимает модель и возвращает модель.

## Сервисы

Сервисы находятся в `src/services`.

Все сервисы должны наследовать `AbstractService`, который содержит большинство методов.

Базовый сервис выглядит так:

```javascript
import AbstractService from './abstractService'
import ProjectModel from '../models/project'

export default class ProjectService extends AbstractService {
	constructor() {
		super({
			getAll: '/projects',
			get: '/projects/{id}',
			create: '/namespaces/{namespaceID}/projects',
			update: '/projects/{id}',
			delete: '/projects/{id}',
		})
	}
	
	modelFactory(data) {
		return new ProjectModel(data)
	}
}
```

Конструктор вызывает родительский и передаёт пути для запросов. Родительский конструктор сохраняет их в сервисе. Все пути опциональны. Вызов метода без определённого пути упадёт с ошибкой.

Placeholder-значения в URL заменяются содержимым переменных с таким же именем в соответствующей модели (той, что передаётся в функции).

### Запросы

Доступные типы запросов:

| Имя | HTTP-метод |
|-----|------------|
| `get` | `GET` |
| `getAll` | `GET` |
| `create` | `PUT` |
| `update` | `POST` |
| `delete` | `DELETE` | 

Каждый метод может принимать модель и опциональные URL-параметры. Кроме `getAll()`, модель обязательна, параметры — нет.

Каждый метод возвращает promise, поэтому результат запроса можно обработать так:

```javascript
service.getAll().then(result => {
	// Делаем что-то с result
})
```

Результат — готовая к использованию модель, созданная фабрикой.

#### Loading

У каждого сервиса есть свойство `loading`, предоставляемое `AbstractModel`. Это `boolean`, автоматически устанавливается в `true` (с задержкой 100 мс, чтобы избежать мерцания) при старте запроса и в `false` по завершении. Используй его для показа/скрытия индикатора загрузки.

### Фабрики

`modelFactory` принимает данные и возвращает модель. Результат всех запросов, кроме `delete`, прогоняется через эту фабрику. Фабрика должна возвращать подходящую модель.

`getAll()` проверяет, является ли ответ массивом, и если да — прогоняет каждый элемент через `modelFactory`.

Можно задать отдельную фабрику для каждого типа запроса, реализовав метод `model{TYPE}Factory(data)` в сервисе. Если специфичная фабрика не найдена, используется `modelFactory`.

### Before Request

Для каждого запроса существует метод `before{TYPE}(model)`. Он получает модель, может её изменить и должен вернуть изменённую версию.

Полезно, например, чтобы превратить JS-даты в unix timestamps.

### After Request?

Нет метода `after{TYPE}`, который вызывался бы после запроса. Обработку сырых API-данных нужно делать в конструкторе модели.

## Модели

Модели проще сервисов. Обычно состоят из объявления значений по умолчанию и опционального конструктора.

Модели находятся в `src/models`.

Каждая модель должна расширять `AbstractModel`. Он обрабатывает парсинг значений по умолчанию.

Модель **не** делает HTTP-запросов — для этого есть сервисы.

Простая модель выглядит так:

```javascript
import AbstractModel from './abstractModel'
import TaskModel from './task'
import UserModel from './user'

export default class ProjectModel extends AbstractModel {
	
	constructor(data) {
		// Конструктор AbstractModel обрабатывает весь парсинг дефолтов.
		super(data)
		
		// Превращаем все задачи в TaskModel
		this.tasks = this.tasks.map(t => {
			return new TaskModel(t)
		})
		
		this.owner = new UserModel(this.owner)
	}
	
	// Дефолтные атрибуты, определяющие "пустое" состояние.
	defaults() {
		return {
			id: 0,
			title: '',
			description: '',
			owner: UserModel,
			tasks: [],
			namespaceID: 0,
			
			created: 0,
			updated: 0,
		}
	}
}
```

### Значения по умолчанию

Функция `defaults()` предоставляет все значения по умолчанию. Конструктор `AbstractModel` берёт переданные данные и заполняет любые отсутствующие, `undefined` или `null` значения дефолтами.

### Конструктор

Конструктор `AbstractModel` обрабатывает парсинг значений по умолчанию. В модели конструктор может делать дополнительную обработку: превращать unix timestamps в JS-даты или парсить дочерний массив в модели.

Если модель ничего такого не делает, конструктор можно не определять — родитель справится сам.

### Доступ к данным

После инициализации модели все свойства доступны через `model.property`. Чтобы свойство точно существовало, укажи его в `defaults()`.
