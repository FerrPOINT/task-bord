# Инструкции для AGENT

## Обзор проекта

Task Board — self-hosted Kanban-приложение с фронтендом на Vue 3 + Vite и бэкендом на Go. Фокус на проектных досках, задачах и командной работе.

Проект состоит из:
- `pkg/` — Go-код API-сервиса
- `frontend/` — веб-клиент на Vue 3
- `Dockerfile` / `docker-compose.yml` — локальная сборка и запуск

## Политика версий API — `/api/v1`

Публичный API живёт в `/api/v1`. Все роуты регистрируются в `pkg/routes/api/v1/` через Huma.

## Навыки

Перед работой в этих областями загрузи соответствующий навык через инструмент `Skill`.

- Добавление или изменение модели в `pkg/models/` (новый CRUD, новые/изменённые `Can*`, работа с правами): загрузи `crudable`.
- Создание или редактирование любого файла в `pkg/migration/`: загрузи `migration`.
- Добавление нового API-роута — всё новое идёт в `/api/v1`, редактирование `pkg/routes/api/v1/`: загрузи `api-v1-routes`.

## Планы и ворктри

Когда пользователь просит создать план для исправления или реализации чего-то:

- ВСЕГДА пиши план в директорию `plans/` в корне репозитория.
- НИКОГДА не коммить планы в git.
- Давай плану описательное имя в kebab-case (например, `fix-position-healing.md`, `feat-new-feature.md`).

### Подготовка ворктри для реализации

Когда пользователь просит подготовить ворктри для плана, используй команду mage для создания изолированного рабочего пространства:

```bash
mage dev:prepare-worktree <name> <plan-path>
```

**Аргументы:**
- `<name>` — обязательный. Становится и именем папки, и именем ветки. Используй конвенции вроде `fix-<описание>` для багфиксов или `feat-<описание>` для фич.
- `<plan-path>` — обязательный. Путь к файлу плана (относительно корня репозитория), который будет скопирован в директорию `plans/` нового ворктри. Передай `""`, чтобы пропустить копирование.

Это создаст новый ворктри в родительской директории и скопирует некоторые файлы.

**Пример:**

```bash
# Создать ворктри для багфикса с планом
mage dev:prepare-worktree fix-position-healing plans/fix-position-healing.md

# Создать ворктри для новой фичи без плана
mage dev:prepare-worktree feat-dark-mode ""
```

**Результат:**

```
parent-directory/
├── main/                    # Оригинальное рабочее пространство
├── fix-position-healing/    # Новый ворктри
│   ├── config.yml           # С обновлённым rootpath
│   └── plans/
│       └── fix-position-healing.md
└── ...
```

После создания скажи пользователю, где находится новый ворктри.

## Команды разработки

### Backend (Go)
- **Сборка**: `mage build` — собирает Go-бинарник
- **Feature-тесты**: `mage test:feature`
- **Web-тесты**: `mage test:web`
- Конкретные тесты: `mage test:filter <filter>` — строка фильтра для go test.
- **Линтер**: `mage lint` — запускает golangci-lint
- **Линтер с автофиксом**: `mage lint:fix`
- **Генерация Swagger**: `mage generate:swagger-docs` — обновляет API-документацию (обычно не нужно, обновляется автоматически в CI)
- **Проверка Swagger**: `mage check:got-swag` — проверяет, что swagger актуален
- **Очистка**: `mage build:clean` — удаляет артефакты сборки
- **Форматирование**: `mage fmt` — форматирует Go перед коммитом

**ВАЖНО:** Для запуска API-тестов обязательно используй `mage test:web`, `mage test:feature` или `mage test:filter`. Обычный `go test` не сработает!

**Go-советы:**
- Для просмотра исходников зависимости или вопросов о ней используй `go mod download -json MODULE` и путь `Dir` для чтения файлов.
- Используй `go doc foo.Bar` или `go doc -all foo` для чтения документации пакетов, типов, функций и т.д.

Хелперы разработки в namespace `dev`:
- **Миграция**: `mage dev:make-migration <StructName>` — создаёт новую миграцию БД. Если опустить `<StructName>`, команда запросит его.
- **Событие**: `mage dev:make-event` — создаёт тип события
- **Listener**: `mage dev:make-listener` — создаёт слушателя событий
- **Уведомление**: `mage dev:make-notification` — скелет уведомления
- **Подготовка ворктри**: `mage dev:prepare-worktree <name> <plan-path>` — создаёт git-ворктри в `../` с заданным именем папки и ветки. Копирует файл плана, если передан. Копирует `config.yml` с обновлённым rootpath и инициализирует фронтенд.

### Frontend (Vue.js)
Перейди в директорию `frontend/`:
- **Dev-сервер**: `pnpm dev` — запускает сервер разработки на порту 4173 (или другом через `--port`)
- **Сборка**: `pnpm build` — production-сборка
- **Dev-сборка**: `pnpm build:dev` — development-сборка
- **Линтер**: `pnpm lint` — ESLint
- **Линтер с автофиксом**: `pnpm lint:fix`
- **Линтер стилей**: `pnpm lint:styles` — Stylelint для CSS/SCSS
- **Автофикс стилей**: `pnpm lint:styles:fix`
- **Type check**: `pnpm typecheck` — проверка Vue TypeScript
- **Unit-тесты**: `pnpm test:unit` — Vitest
- **E2E-тесты**: НЕ запускай `pnpm test:e2e` напрямую. Используй `mage test:e2e` (см. ниже).

### Pre-commit проверки
Перед коммитом всегда запускай линтер:

```bash
# Backend
mage lint:fix

# Frontend  
cd frontend && pnpm lint:fix && pnpm lint:styles:fix
```

Исправь ошибки, которые они сообщат, и только потом коммить.

Запускай линтер бэкенда только при изменении backend-кода, а frontend-линтер — только при изменении frontend-кода. Stylelint — только при правке CSS/SCSS или стилей Vue-компонентов.

## Обзор архитектуры

### Backend Architecture (Go)
Бэкенд на Go построен по слоистой архитектуре с чётким разделением ответственности:

**Core Layers:**
- **Models** (`pkg/models/`) — доменные сущности с бизнес-логикой и CRUD-операциями
- **Services** (`pkg/services/`) — слой бизнес-логики для сложных операций
- **Routes** (`pkg/routes/`) — HTTP API endpoints и роутинг
- **Web** (`pkg/web/`) — универсальные CRUD-хендлеры и абстракции веб-фреймворка

**Ключевые паттерны:**
- **Generic CRUD**: модели реализуют интерфейс `CRUDable` для стандартных операций с БД
- **Permissions System**: трёхуровневая система прав (Read/Write/Admin) на всех операциях
- **Event-Driven**: событийная система для уведомлений и сквозной логики
- **Modular Design**: подключаемая аутентификация, провайдеры аватаров, инструменты миграции

**База данных:**
- ORM XORM с PostgreSQL (production) и SQLite (тесты)
- Система миграций в `pkg/migration/` с файлами по временным меткам
- Сессии БД с автоматическим управлением транзакциями

**Аутентификация:**
- Локальная аутентификация по паролю
- JWT-токены для доступа к API

### Frontend Architecture (Vue 3)
Современное приложение на Vue 3 Composition API с TypeScript:

**State Management:**
- **Pinia** сторы в `src/stores/` для глобального состояния
- Composables в `src/composables/` для переиспользуемой логики
- Компонентный уровень состояния через Vue 3 Composition API

**Ключевые директории:**
- `src/components/` — переиспользуемые Vue-компоненты, сгруппированные по фичам
- `src/views/` — страничные компоненты и роутинг
- `src/stores/` — Pinia state management
- `src/services/` — API-сервисный слой, соответствующий backend-моделям
- `src/models/` — TypeScript-интерфейсы, соответствующие backend-моделям
- `src/helpers/` — утилиты и бизнес-логика

**UI Framework:**
- CSS-фреймворк Bulma с CSS-переменными для темизации
- FontAwesome-иконки с tree-shaking
- Rich text редактор TipTap для описаний задач
- Собственная библиотека компонентов в `src/components/base/`

## Рабочие процессы разработки

### Добавление новых фич

**Backend:**
1. Создай/измени модели в `pkg/models/` с нужными CRUD и Permissions-интерфейсами (загрузи навык `crudable`)
2. Добавь миграцию БД, если нужно: `mage dev:make-migration <StructName>` (загрузи навык `migration`)
3. Создай/обнови сервисы в `pkg/services/` для сложной бизнес-логики
4. Добавь API-роуты в `/api/v1` в `pkg/routes/api/v1/` — загрузи навык `api-v1-routes`.

**Frontend:**
1. Создай TypeScript-интерфейсы в `src/modelTypes/` по backend-моделям
2. Добавь/обнови сервисы в `src/services/` для общения с API
3. Создавай компоненты в соответствующих поддиректориях `src/components/`
4. Добавляй страницы в `src/views/` с правильным роутингом
5. Обновляй Pinia-сторы, если изменяется глобальное состояние

### Изменения базы данных
1. Запусти `mage dev:make-migration <StructName>`
2. Отредактируй сгенерированный файл миграции в `pkg/migration/`
3. Обнови соответствующую модель в `pkg/models/`
4. Обнови TypeScript-интерфейсы во frontend `src/modelTypes/`

### API Development
- Все эндпоинты идут в `/api/v1` (`pkg/routes/api/v1/`) через Huma.
- Используй REST-конвенции: POST создаёт, PUT/PATCH обновляют, DELETE удаляет.
- Переиспользуй универсальные `Do*`-функции из `pkg/web/handler/`, которые применяют права через `Can*`-методы моделей.
- Проверяй права на уровне модели через Permissions interface — никогда в роут-хендлере (исключение: non-CRUD действия должны явно вызывать `Can*`).
- OpenAPI-спека генерируется из Go-типов Huma автоматически.

### Тестирование
- Backend: feature-тесты рядом с исходниками, web-тесты в `pkg/webtests/`
- Frontend: unit-тесты на Vitest, E2E-тесты на Playwright
- Всегда тестируй и позитивные, и негативные сценарии авторизации
- Используй фикстуры в `pkg/db/fixtures/` для консистентных тестовых данных

### Запуск E2E-тестов

**ВАЖНО: ВСЕГДА используй `mage test:e2e` для end-to-end тестов.** Не запускай `pnpm test:e2e` напрямую. Команда mage собирает API, запускает его с изолированной SQLite-базой, собирает и раздаёт фронтенд, запускает Playwright и затем всё останавливает.

```bash
mage test:e2e ""                                      # все тесты
mage test:e2e "tests/e2e/misc/menu.spec.ts"           # конкретный файл
mage test:e2e "--grep menu"                            # фильтр по имени
mage test:e2e "--headed tests/e2e/misc/menu.spec.ts"  # headed-режим
```

**ВАЖНО: Всегда сохраняй вывод тестов в файл.** E2E-тесты дорогие (пересобирают API, запускают серверы, браузеры и т.д.). НИКОГДА не перезапускай тесты только для того, чтобы посмотреть вывод по-другому (с другим `grep`/`tail`). Вместо этого сохраняй вывод при первом запуске, а потом читай файл:

```bash
# Первый запуск: сохраняем в файл
mage test:e2e "tests/e2e/misc/menu.spec.ts" 2>&1 | tee /tmp/e2e-output.log

# Анализ: читаем файл, не перезапускаем
# ...
```

То же касается `mage test:web`, `mage test:feature` и `mage test:filter`.

Установи `TASKBOARD_E2E_SKIP_BUILD=true`, чтобы пропустить пересборку API-бинарника при итерациях только по frontend.

## Swagger API Documentation

Никогда не трогай сгенерированную swagger-документацию в `pkg/swagger/`. Она автоматически генерируется CI после коммитов.

## Сообщения коммитов

Используй стиль **Conventional Commits** (например, `feat: add foo` или `fix: correct bar`). Репозиторий использует эти сообщения для генерации changelogs.

## Frontend Development Guidelines

Веб-клиент живёт в `frontend/` и использует Vue 3 + TypeScript. ESLint требует: одинарные кавычки, trailing commas, без точек с запятой, отступы табом, Vue `<script lang="ts">`, PascalCase для компонентов, camelCase для событий. Смотри `frontend/eslint.config.js` и `frontend/.editorconfig` и следуй правилам форматирования там.

## Переводы

При добавлении или изменении функциональности, которая затрагивает пользовательские сообщения, их нужно переводить.

Во frontend все строки перевода живут в `frontend/src/i18n/lang`. Для API (в основном локализация уведомлений) строки живут в `pkg/i18n/lang`.

Нужно править только файл `en.json` с исходной строкой. Сам перевод происходит в другом месте.
После изменения исходной строки нужно вызвать соответствующую библиотеку перевода с ключом. Обе похожи, смотри существующий код.

**Не добавляй новый язык с нуля и не переводи строки на другие языки сам.** Переводы управляются отдельным workflow. Если тебя просят добавить язык, перевести существующие строки или обновить переводы для non-English локалей — укажи на гайд по переводам.

## Ключевые файлы и конвенции

**Конфигурация:**
- `config-raw.json` — источник структуры конфигурации
- Переменные окружения переопределяют настройки файла конфигурации
- Для управления конфигурацией используй `pkg/config/`

**Code Style:**
- Go: golangci-lint по `.golangci.yml`; используй goimports; оборачивай ошибки через `fmt.Errorf("...: %w", err)`; проверяй права в моделях; никогда не логируй секреты; не редактируй сгенерированные `pkg/swagger/*`
- Vue: ESLint + TS; одинарные кавычки, trailing commas, без точек с запятой, таб; script setup + lang ts; держи сервисы/модели в синхроне с backend
- Следуй существующим паттернам
- **Комментарии: документируй *почему*, а не *что* — по умолчанию без комментариев.** Не пиши комментарии, которые повторяют код, имя функции/структуры/поля или сигнатуру; это шум, который читатель пропускает (комментарий, который дольше читать, чем код, должен быть удалён). Комментируй только действительно нетривиальное *почему* — подводный камень, инвариант, отвергнутую альтернативу, ограничение между файлами — одной ёмкой строкой. Будь агрессивен в удалении на первом проходе, не только когда попросят.
- Перед созданием нового файла, функции или хелпера ищи в кодовой базе (`grep` / `rg`) существующий код, который делает то же самое. Предпочитай расширить существующий хелпер, а не дублировать. Если логика существенно пересекается с существующей функцией — переиспользуй её.

**Naming Conventions:**
- Go: стандартные Go-конвенции (PascalCase для экспорта, camelCase для приватного)
- Vue: PascalCase для компонентов, camelCase для composables
- API endpoints: kebab-case в URL, snake_case в JSON

**Права доступа:**
- Всегда реализуй Permissions interface для новых моделей
- Используй методы `CanRead`, `CanWrite`, `CanCreate`, `CanDelete`
- Права проверяются на уровне модели, не только роутов

## Типичные подводные камни

- Миграции БД необратимы в production — тщательно тестируй
- Frontend-сервисы должны точно соответствовать структуре backend-моделей
- Проверка прав обязательна для всех CRUD-операций
- Слушатели событий в `pkg/*/listeners.go` должны быть корректно зарегистрированы
- CORS-настройки backend должны разрешать домен frontend

## Scope Notes

The following upstream subsystems have been removed from this fork and should not be reintroduced without an explicit user request:
- CalDAV, Webhooks, Audit logging, Sentry, Redis-backed queues, API tokens, Plugins/Yaegi, Time tracking, Link sharing, TOTP, OAuth/OpenID/LDAP auth, Data migrators, Unsplash/Gravatar/Marble integrations, License/feature gates.
